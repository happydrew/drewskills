# Git Commit And Push

你的任务是在当前工作目录中，把用户已经完成的代码改动提交到当前 Git 分支，并推送到对应远端。

适用场景：

- 用户明确要求“提交代码”“提交当前项目代码”“提交代码到仓库”“把改动推上去”等
- 当前目录是一个 Git 仓库，且用户意图是提交本地已有改动，而不是继续开发新功能
- 用户没有指定要拆分多个提交，而是希望一次完成当前改动的提交和推送

不适用场景：

- 用户只想查看改动、生成 commit message 草稿或做 code review，还没有要求实际提交
- 当前目录不是 Git 仓库，或仓库没有配置可推送的远端分支
- 工作区存在明显不应一起提交的无关改动，且无法安全判断提交边界
- 用户明确要求不要 push，或只要求本地 commit

需要先检查的信息：

- 当前目录是否为 Git 仓库
- 当前是否位于某个分支上，而不是 detached HEAD
- 工作区是否存在可提交改动
- 当前分支是否已配置上游远端；若没有，需要显式告诉用户无法直接 push
- 是否存在明显冲突、合并中状态或其他会阻塞提交的 Git 状态

工作流程：

1. 先查看 `git status --short --branch`，确认仓库、分支、改动范围和是否存在异常状态
2. 如果没有任何可提交改动，直接告诉用户当前没有需要提交的内容，不要继续执行
3. 如存在会影响安全提交的问题，例如 detached HEAD、merge 冲突、rebase 未完成、没有上游分支，先停止并明确说明原因
4. 默认在当前目录执行 `git add ./`
5. 基于当前改动生成一条简明扼要的 commit message：
   - 优先概括本次改动的核心目的
   - 尽量短，不写空话，不堆叠多个不相关主题
   - 默认使用英文 Git 风格短句，例如 `feat: add xxx`、`fix: correct xxx`、`docs: update xxx`
6. 执行 `git commit -m "<message>"`
7. 执行 `git push`，推送到当前分支的已配置上游
8. 返回提交结果，明确写出分支名、commit message、是否 push 成功

输出要求：

- 直接说明是否已完成 add、commit、push
- 明确写出当前分支名和最终使用的 commit message
- 如果失败，明确写出失败发生在检查、commit 还是 push 阶段
- 如果因为 Git 状态不满足而未执行，明确告诉用户缺少什么条件

禁止事项：

- 不要在不是 Git 仓库的目录里盲目执行提交命令
- 不要在 detached HEAD、冲突未解决或 rebase/merge 未完成时强行提交
- 不要臆造 commit message 已生成或 push 已成功
- 不要擅自切换分支、创建分支、拉取远端或重写历史
